## ⚠️ 重要な注意事項

### 成果物形式の必須ルール

**テストケースは必ずTSV形式で作成してください**:
- テンプレート: [`成果物テンプレート/テンプレート_テストケース一覧.tsv`](../../../成果物テンプレート/テンプレート_テストケース一覧.tsv)
- ファイル名規約: `{PROGRAM_NAME}_テストケース.tsv`
- Markdown形式は議論・レビュー段階のみ使用可能
- 最終成果物は必ずTSV形式で提出

### テスト観点の限定列挙
 
本テンプレートで使用するテスト観点は以下の3つに限定されます（自由記述で観点名を作らないでください）。
 
| 観点 | 内容 |
| :--- | :--- |
| 条件分岐網羅 | if/switch文の真偽両ルート、三項演算子、論理演算子の短絡評価、関数戻り値のチェック分岐。ファイル入出力・メッセージ出力・0/1/N件チェック・ループ処理に加え、**C言語特有のメモリ管理（malloc/free）・ポインタ操作（NULLチェック）・関数の戻り値判定（fopen/malloc等）もすべてこの観点に含める** |
| 境界値 | 閾値±1、入力件数・カウンタの0/1/N |
| シーケンスチェック | キー項目の昇順・降順・マッチングパターン |
 
SQL関連の観点（SQL観点: ORDER BY/GROUP BY、区分判定、SQL実行結果、データ有無、コミット結果等）は本テンプレートの対象外です。DB処理を含むプログラムの場合は、別途SQL用プロンプトテンプレート（`プロンプトテンプレート/単体テストケース/SQL/`）を併用してください。

**テスト観点のアドバイス**

* **網羅性**
  - すべての条件分岐（if / else、switch-case）について、条件が成立する場合（真）と成立しない場合（偽）の両方のルートをテストケースに含めている。init()のファイルオープン成否、read_record()のEOF・読み込みエラー判定、validate_data()のデータ検証成否、write_record()の書き込み成否、cleanup()でのポインタNULLチェック（free/fclose実行有無）を、すべて「条件分岐網羅」の観点として網羅している。
  - malloc/freeによるメモリ管理、ポインタのNULLチェック、fopen()・malloc()等の戻り値判定についても、独立観点にはせず「条件分岐網羅」の一種として扱い、テストケース・期待結果の記述内でどの分岐に該当するかを明確にしている（例: TC-002はfopen()戻り値判定、TC-003はmalloc()戻り値判定、TC-015/016はポインタNULLチェック分岐）。
  - 分岐に至るまでの他の条件因子（例: TC-008の「データ長がMIN_LENGTH未満」は、そこに至る前段のファイルオープン成功・レコード読み込み成功という条件因子を前提とする）も踏まえてテストケースを作成している。
  - 入力ファイルの件数パターン（0件、1件、複数件）による挙動の違いを確認するケース（TC-019, TC-020等）を含めている。
  - メッセージ一覧表に開始・終了メッセージの出力仕様が明記されている場合は、それらが正しく出力されることを確認するケース（TC-001, TC-014）を含めている。指示書に該当仕様がない場合は、無理にテストケース化していない。

* **境界値**
  - データ長チェックのように数値比較を伴う条件分岐では、閾値そのものだけでなく閾値±1（例: `strlen(buffer) = MIN_LENGTH - 1`と`MIN_LENGTH`）をテストケースに含めている（TC-008）。
  - 入力ファイルの件数・カウンタについても、0件・1件・N件（複数件、大量件数）の境界パターンを確認している（TC-019, TC-020）。
  - 指示書中に他の数値閾値（リトライ回数の上限、バッファサイズの上限など）が存在する場合は、同様に閾値±1のパターンを追加で検討すること。

* **シーケンスチェック**
  - シーケンスチェックは、指示書に記載されたキー項目（例: NEWKEYとOLDKEYなど）の比較によって、昇順・降順の並び順や、キーの一致・不一致（マッチング）パターンを確認する観点である。
  - 具体的には、「新旧キーが一致する場合」「新キーが旧キーより大きい（正しい昇順）場合」「新キーが旧キーより小さい（シーケンスエラー）場合」のように、比較結果ごとに分岐する処理をテストケース化する。
  - 本サンプルプログラムの指示書にはキー項目の突合・整列処理が確認できないため、該当するテストケースは作成していない。対象プログラムにマスタ突合やキー順チェックの処理がある場合は、上記の観点に沿ってテストケースを追加すること。

* **異常系**
  - ファイルオープン失敗（TC-002）、ファイル読み込みエラー（TC-006）、ファイル書き込みエラー（TC-011）、メモリ確保失敗（TC-003）、データ検証失敗（TC-008, TC-009）、init()失敗によるプログラム異常終了（TC-018）など、プログラムがエラー終了またはエラーログ出力するパスをすべてテストケースに含めている。
  - 各異常系では、対応するエラーメッセージ（メッセージ一覧表に記載の文言）が標準エラー出力に正しく表示されることを、期待結果に明記している。
  - SQL処理やデータベース関連の戻り値判定は本テンプレートの対象外のため、テストケース化していない（該当する場合は別途SQL用テンプレートを使用する）。

---

**テスト実施上のポイント**

* テストデータを準備する際は、対象のレコードのどの項目値を変えれば「検証成功」または「検証失敗」の分岐に入るかを事前に整理し、それぞれの分岐を通るデータをファイル単位で用意する（例: TC-007は必須フィールドに値が入ったレコード、TC-009は必須フィールドが空のレコードを用意する）。
* 期待結果は外部から観測可能な結果（標準出力・標準エラー出力のメッセージ、出力ファイルのレコード内容、プログラムの戻り値）に限定して記載し、内部変数やフラグの状態は記載しない（指示書「3. 期待結果の定義」を参照）。

これらの観点を満たすテストケースを実行することで、C言語プログラムの主要ロジック、エラーハンドリング、メモリ管理、ファイル操作の正確性を網羅的に検証できます。
