# C言語単体テストケース生成

## 概要

C言語プログラム指示書を入力として、単体テストケースを生成するスキル。
正常系・異常系・境界値を網羅したテストケースと、テスト観点のアドバイスを生成し、最終成果物として「テストケース（TSV）」と「テストケース一覧＋テスト観点のアドバイス（Markdown）」の2点を出力する。
本スキルは単発（このスキル単体）でも呼び出せるように設計しており、モード遷移の制御は行わない。

遵守事項・制約・禁止事項は [`.bob/rules-c-testcase-designer/AGENTS.md`](../../rules-c-testcase-designer/AGENTS.md) を正本とする。

## 実行手順

以下の手順を順番に実行してください。ルールの詳細は都度 AGENTS.md を参照すること。

### Step 1: セッション記憶の開始

[`MEMORY/`](../../../MEMORY/) 配下にセッションログを作成する。
ファイル名: `MEMORY/sessions/YYYY-MM-DD_session-testcase-NNN.md`（当日既存の最大番号 +1）

### Step 2: 入力ファイルの読み込み

**読み込み前に必ず `list_files` で存在確認してから `read_file` を実行すること。**

引数でディレクトリが指定された場合はそのディレクトリ、指定がない場合は `インプット/` をデフォルトとする。

以下の順序で読み込む:

1. **プログラム指示書**（Markdown形式）— 必須
   - `インプット/` 配下の対象 `.md` ファイル
   - ファイル内の画像リンクも確認すること
2. **セルフチェックリスト**（Markdown形式）— 必須
   - `セルフチェックリスト/` 配下の単体テストケースチェックリスト
3. **成果物テンプレート**（TSV形式）— 必須
   - `成果物テンプレート/` 配下の単体テストケーステンプレート
4. **プロンプトテンプレート** — 必須
   - `プロンプトテンプレート/単体テストケース/` 配下（存在する場合）
5. **設計規約** — オプション
   - `インプット/` 配下の設計規約 `.md` ファイル（存在する場合）
6. **参考資産** — 推奨
   - `参考資産/` 配下の単体テストケース生成例（存在する場合）

### Step 3: DB処理の有無を判定

プログラム指示書から以下のキーワードを検索し、使用するプロンプトテンプレートを判定する。
 
**DB 処理判定キーワード**: `SQL`, `sqlite3`, `mysql`, `postgresql`, `データベース`, `DB接続`
 
| 判定結果 | 使用テンプレート |
|:---|:---|
| キーワードなし（DB 処理なし） | `プロンプトテンプレート/単体テストケース/通常/` のみ |
| キーワードあり（DB 処理あり） | `通常/` と `SQL/` の両方 |
 
DB処理ありと判定した場合、テスト観点「SQL観点」（**テストケース生成**で後述）も適用対象とする。
### Step 4: テストケース生成

プロンプトテンプレートの `UserPrompt.txt` を読み込み、`{program_instruction}` を置換する（変数置換ルールは`AGENTS.md`参照）。
 
UserPromptの指示に従い、以下の2つを生成する:
1. **テストケース**: Markdownテーブル形式で生成する。この段階のMarkdownは、レビューに必要な観点・情報を過不足なく含んだ**中間成果物**とする。列順序・列選択を最終フォーマット（TSV）に合わせる必要はなく、必要な情報を網羅することを優先する。最終フォーマットへの並べ替え・取捨選択は`成果物テンプレート/`への変換時に行う（`AGENTS.md`**出力構成（列定義）の管理方針**参照）
2. **テスト観点のアドバイス**: テストケーステーブルの後に、UserPromptで指示された観点（網羅性・境界値・シーケンスチェック・異常系 等）に基づき箇条書きで生成する

**仕様が曖昧な場合:**
- 推測で補完しない
- 仮定を列挙して「不明」として明示する
- 生成後にユーザーレビューを前提とする

#### C言語のテスト観点（限定列挙）
 
テスト観点は以下に限定する。これ以外の自由記述の観点名を作らないこと。
 
| 観点 | 内容 |
|:---|:---|
| 条件分岐網羅（C2） | 「C2」は条件文・分岐の真偽をすべて通すことを狙うカバレッジ観点の呼称（作業者が用いる用語のため名称を残す）。具体的には、if/switch文の真偽両ルート、三項演算子の真/偽、論理演算子（&&, \|\|）の短絡評価、関数戻り値のチェック分岐（rc != 0 等）。ファイル入出力・メッセージ出力・0/1/N件チェック・ループ処理もこの観点に内包する。以下もすべて条件分岐網羅に属するものとして扱う（テストケース記載時は観点名を分けず「条件分岐網羅(C2)」のまま、テストケース内容側でどの分岐かを明記する）：<br>・メモリ管理にあたるもの: malloc/freeの確保・解放結果判定の分岐<br>・ポインタ操作にあたるもの: NULLポインタチェックの分岐<br>・関数の戻り値判定にあたるもの: fopen/malloc等の戻り値異常時のif分岐、異常終了処理の実行分岐（**SQL実行結果・区分判定・コミット結果などSQLに関する判定分岐はSQL観点に含め、条件分岐網羅には含めない**） |
| 境界値 | 閾値±1、入力件数・カウンタの0/1/N、定数値（コミット件数等）の前後を網羅する。|
| シーケンスチェック | キー項目の昇順・降順・マッチングパターン（=/>/<の組合せ）、ファイル中身がソートされているかを確認する（ファイル・データ構造上のキー順序が対象） |
| SQL観点（DB処理ありの場合のみ） | SQL実行に関する判定・分岐はすべてこの観点に含める（条件分岐網羅には含めない）。具体的には：区分判定（各区分値）の分岐、SQL実行結果（成功/失敗）の判定分岐、データ取得結果（データあり/データなし）の判定分岐、コミット実行結果（成功/失敗）の判定分岐、ORDER BY句による並び順（昇順・降順）、GROUP BY句によるグルーピング結果（グループ単位の集計・件数）。抽出条件の粒度ルール（複数項目から成る条件は「全項目一致」「いずれか不一致」の2ケース）やデータイメージ確認などの具体的な生成ルールは、SQL用UserPrompt.txt（`プロンプトテンプレート/単体テストケース/SQL/`）を正本とする |

#### 関数種別ごとの観点適用例（参考・網羅性確認用）
 
各観点の内容を繰り返さず、どの関数種別にどの観点を重点適用するかの対応のみを示す。実際のテストケース作成では上記の各観点の定義を参照すること。「条件分岐網羅」「SQL観点」欄の（）内は、該当する補足内容を示す。**SQL実行関数（SQLを直接発行する関数）は、戻り値・結果判定であっても条件分岐網羅ではなくすべてSQL観点として扱う。**
 
| 関数種別 | 条件分岐網羅 | 境界値 | シーケンスチェック | SQL観点 | 補足（対象処理） |
|:---|:---|:---|:---|:---|:---|
| 初期化関数 | ○（関数の戻り値判定：ファイルオープン、DB接続） | - | - | - | 環境変数取得、区分判定、ファイルオープン、DB接続 |
| 入力ファイル読込関数 | ○ | - | - | - | 読込件数0/1/複数件 |
| メイン処理関数 | ○（関数の戻り値判定） | ○ | - | - | 入力件数境界値、サブ関数戻り値、読込エラー |
| SQL実行関数（TRUNCATE/DELETE） | - | - | - | ○（区分判定、SQL実行結果） | 区分判定、SQL実行結果 |
| SQL実行関数（SELECT/ORDER BY/GROUP BY） | - | - | - | ○（SQL実行結果、データ有無、ORDER BY、GROUP BY） | SQL実行結果、データ有無、並び順・グルーピング |
| SQL実行関数（INSERT/UPDATE） | - | ○（コミット件数境界値） | - | ○（区分判定、SQL実行結果、コミット結果） | 区分判定、SQL実行結果、コミット結果、コミット件数境界値 |
| 正常終了関数 | ○（関数の戻り値判定：ファイルクローズ） | - | - | ○（コミット結果） | ファイルクローズ、コミット成否 |
| 異常終了関数 | ○（関数の戻り値判定） | - | - | - | 異常終了処理の実行確認 |
| ポインタを扱う関数全般 | ○（メモリ管理／ポインタ操作） | - | - | - | メモリ確保・解放、NULLポインタ |

### Step 5: AIによる第三者チェックと修正（生成後必須）

生成直後、ユーザーレビュー前に第三者の視点でチェックを行う。

`AGENTS.md`**セルフチェックの実施タイミング**に従い、テストケース生成直後・ユーザーレビュー前に必ず実施する（省略禁止）。手順は以下の通り:
 
1. `セルフチェックリスト/` 配下のチェックリストに基づき、不足・誤りを特定する
2. 修正案を提示する（**修正理由を必ず明記すること**）
3. 修正後のテストケースを出力する
4. 修正内容のサマリーを提供する

**チェック観点（主要項目）**:
- 本ファイル**テストケース生成**の観点（条件分岐網羅・境界値・シーケンスチェック・SQL観点）が過不足なく使われているか
- 正常系・異常系・境界値がすべて含まれているか
- テスト観点が限定列挙内か（自由記述で観点名を作らない）
- 項目に過不足がないか（成果物テンプレートの列構成と照らして確認）
- IDが一貫しているか
- 期待結果が外部観測可能な振る舞いのみか
- テスト観点のアドバイスが、生成したテストケースの内容と矛盾していないか
- **（DB処理ありの場合）** `SQL/UserPrompt.txt`の指示（抽出条件の粒度ルール：複数項目から成る条件は「全項目一致」「いずれか不一致」の2ケース、データイメージ確認）が満たされているか

### Step 6: 最終成果物の出力（TSV + Markdown）
 
`AGENTS.md`**「成果物の形式」「出力先・ファイル名**」に従い、以下の2ファイルを出力する。
 
1. **テストケース本体（TSV形式・必須）**
   - `成果物テンプレート/` のTSVテンプレートに沿って、**テストケース生成**で生成したテストケーステーブルを変換する
   - 列構成・列順序は`成果物テンプレート/`のテンプレートに従ってセルを並べる（中間成果物のMarkdownテーブルと列順序が一致しなくてよい）。タブ文字でセルを区切り、ヘッダ行を含めること
   - 出力ファイル: `アウトプット/{PGM_ID}_テストケース.tsv`
2. **テストケース一覧＋テスト観点のアドバイス（Markdown形式・必須）**
   - **AIによる第三者チェックと修正**で確定した内容を **そのままの順序** でMarkdownファイルに保存する（テストケース一覧を単独で省略しないこと）
     1. 「テストケース」見出し＋テストケースのMarkdownテーブル（一覧として参照できるように、TSVとは別に丸ごと保持する）
     2. 「テスト観点のアドバイス」見出し＋箇条書き
   - TSVへの変換・再構成は不要（このファイルはあくまで人が読むための参照用であり、正本はテストケース本体のTSVとする）
   - 出力ファイル: `アウトプット/{PGM_ID}_テスト観点のアドバイス.md`

### Step 7: ユーザー確認

`ask_followup_question` ツールを使用してユーザーに承認可否を確認する（文言の自由記述は禁止）。提示する選択肢の正本は `AGENTS.md`**ユーザー承認確認の実施方法** の選択肢テーブルとし、本ファイルには再掲しない。

- **未承認のまま次工程へ進むことは禁止**

### Step 8: セッション記憶の完了

[`MEMORY/`](../../../MEMORY/) のセッションログを完成させる。
エラーや新しい学びがある場合は `MEMORY/reflections/YYYY-MM-DD_reflection.md` を生成し、
`MEMORY/sessions/index.md` と `MEMORY/reflections/index.md` を更新する。

## 入力制約・成果物形式・次工程の扱い

- 出力先・ファイル名、ユーザー承認、次工程の扱いは `AGENTS.md` を正本とする（本ファイルには重複記載しない）。
