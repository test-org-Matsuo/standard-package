# ショット割り当て

## 概要

テストデータの数を最小化するために、単体テストケースに新規ショット番号を割り当てる。
テストケースの内容を分析し、効率的なグループ化を提案・適用する。

## 目的

- **テストケースの内容を分析**
- **グループ化可能なケースを特定**（同一ショットにまとめられる条件を満たすケース）
- **新規ショット番号を割り当て**
- **テストデータの作成・管理コストを削減**

## 実行手順

**重要**: このコマンドは **Agentモード** で実行する。

### 1. セッション記憶の開始

[`MEMORY/`](../../../MEMORY/) 配下にセッションログを作成する。
ファイル名: `MEMORY/sessions/YYYY-MM-DD_session-assign-NNN.md`（当日既存の最大番号 +1）

### 2. 入力ファイルの確認

**入力ファイル**:
- 単体テストケース（TSV形式）
- プログラム指示書（Markdown形式、オプション）

**確認方法**:
```
# ファイルの存在確認（read_file前に必ず実行）
list_files <ディレクトリパス>
```

### 3. テストケース分析

#### 3.1 テストケースの読み込み

1. **`list_files`で存在確認後、`read_file`で読み込み**
   - 単体テストケース（TSV形式）
   - プログラム指示書（必要に応じて）

2. **テストケースの内容を分析**
   - 各テストケースの処理パターン（正常系/異常系）
   - 入力ファイルの件数パターン
   - 期待結果の整合性
   - 異常終了（非0リターンコード）発生の有無

3. **プログラム指示書から情報を抽出**（必要に応じて）
   - 入力ファイル数・出力ファイル数
   - 異常終了条件・リターンコード定義
   - 0件確認要否

#### 3.2 グループ化可能なケースを特定

以下の観点でグループ化の可能性を特定する：

**同一ショットにまとめられるケース**:
- 入力ファイル件数が同じで、期待結果が矛盾しない正常系ケース
- 複数レコードで複数条件を検証可能な関連ケース
- 同じ処理セクション内で関連する条件

**別ショットにする必要があるケース**:
- 異常系（異常終了・非0リターンコード発生）ケース
- 矛盾する期待結果を持つケース
- 入力ファイル件数が異なるケース

### 4. ショット割り当てルールの適用

**ルール適用の優先順位**: 複数ルール間で競合が発生した場合、記載順に優先度が高いものとして適用する。

#### 4.1 必ず別ショットにする条件

1. **入力ファイル件数の組み合わせケース**:
  - 入力ファイルの件数が0件のケース、1件のケース、N件のケースは別ショットとすること。
  - 入力ファイルが複数ある場合は、その件数の組み合わせパターンは別ショットとすること（下表の例を参照）。
    | ショットNo. | IN01 | IN02 |
    | --- | --- | --- |
    | ST001 | 0件 | 0件 |
    | ST002 | 0件 | 1件 |
    | ST003 | 0件 | N件 |
    | ST004 | 1件 | 0件 |
    | ST005 | 1件 | 1件 |
    | ST006 | 1件 | N件 |
    | ST007 | N件 | 0件 |
    | ST008 | N件 | 1件 |
    | ST009 | N件 | N件 |
2. **異常系ケース（異常終了・非0リターンコード発生）**: 各異常パターンごとに独立したショット
  - 理由: エラー検出時点でプログラムが異常終了するため、1回のテスト実行で1つのエラーパターンしか検証できない
  - 例: ファイルオープンエラーパターンは各パターンごとに独立したショットが必要
3. **境界値テストケース**: 特定の値を検証する場合は独立したショット
  - 例: 配列インデックスの上限値・下限値付近
  - 例: バッファサイズ境界（配列オーバーフローで異常終了するケース）

#### 4.2 同一ショットにまとめられる条件

- **処理パターンが競合しない**
  - 例: 正常系と異常系は別ショット
  - 例: シーケンスチェック正常と異常は別ショット
- **期待結果が矛盾しない**
  - 例: 同じファイルに対して「0件出力」と「N件出力」は別ショット
- **複数レコードで複数条件を検証可能**
  - 異なるケースでも、レコードを増やすことでまとめてテスト可能なものについては、ショット番号を同一にする
  - 例: ソートキー正常パターン（昇順1, 昇順2, ...）は、入力ファイルに複数レコードを含めることで1ショットで検証可能

### 5. ショット割り当て案の提示

テストケース分析とショット割り当てルール適用の結果を以下の形式で提示する：

```markdown
## ショット割り当て方針

### テストケース構成
- 総テストケース数: YY個
- 新規採番するショット数: XX個
- 採番方針: ST001から連番で付番
- グループ化根拠: 入力件数、正常系/異常系、期待結果の整合性

### グループ化の内訳

#### 同一ショットにまとめるケース
- ケース1, 2, 3 → ST001（正常系、入力件数が同じ）

#### 独立したショットとするケース
- ケース4 → ST002（IN01:0件、IN02:0件）
- ケース5 → ST003（IN01:0件、IN02:1件）
- ケース6 → ST004（IN01:0件、IN02:N件）
...
- ケース13 → ST011（境界値テスト）
- ケース14 → ST012（境界値テスト）
- ケース15 → ST013（異常系、非0リターンコード発生）
- ケース16 → ST014（異常系、非0リターンコード発生）

### 具体的な割り当て内容
1. ケース1, 2, 3 → ST001
2. ケース4 → ST002
3. ケース5 → ST003
4. ケース6 → ST004
...
5. ケース13 → ST011
6. ケース14 → ST012
7. ケース15 → ST013
8. ケース16 → ST014
```

### 6. ユーザー承認

ショット割り当て案を提示後、以下を確認する：

```
ショット割り当て案を承認するか？
- 承認する場合: 「承認」と入力
- 修正が必要な場合: 具体的な修正内容を指示
```

### 7. ショット番号の割り当て

ユーザー承認後、以下の作業を実行する：

#### 7.1 テストケースへのショット番号割り当て

1. 元のテストケースTSVファイルを読み込む
2. 承認したショット割り当て案に基づき、各テストケースの項目番号とショット番号を対応付ける
3. 新しいSHOT名（ST001, ST002, ...）を「ショットNo」列に記入
4. 更新されたテストケースを元のファイルパスに上書き保存

**例**:
```tsv
項目番号	関連ルーチン	正常系/異常系	テスト観点	テストケース	期待結果	ショットNo
1	main	正常系	ファイル入出力	ケース1	結果1	ST001
2	main	正常系	条件分岐	ケース2	結果2	ST001
3	sub_func	異常系	条件分岐	ケース3	結果3	ST002
```

### 8. 出力ファイルの保存

以下のファイルを保存する：

**ショット番号が割り当てられたテストケース**:
- 元のテストケースTSVファイルを上書き更新
- 「ショットNo」列にショット番号（ST001, ST002, ...）が記入される

### 9. 最終確認

1. **ショット割り当て結果のサマリー表示**
   - 総ショット数
   - 総テストケース数
   - 平均ケース数/ショット
   - 主な割り当て内容

2. **整合性確認**
   - すべてのテストケースが含まれているか
   - グループ化ルールに従っているか
   - 矛盾するケースが同一ショットに含まれていないか

3. **ファイル保存確認**
   - テストケースTSVファイルが更新されたか

### 10. セッション記憶の完了

[`MEMORY/`](../../../MEMORY/) のセッションログを完成させる。
エラーや新しい学びがある場合は `MEMORY/reflections/YYYY-MM-DD_reflection.md` を生成し、
`MEMORY/sessions/index.md` と `MEMORY/reflections/index.md` を更新する。
リフレクションは `MEMORY/USAGE_GUIDE.md` のテンプレートに厳密に従うこと。

## ショット割り当ての例

### 例1: 正常系のグループ化

**テストケース**:
```
ケース4: 入力ファイルAがN件
ケース5: 入力ファイルBがN件
ケース6: 出力ファイルCの項目内容確認
```

**割り当て結果**:
```
ST003: ケース4, 5, 6（すべて正常系でN件処理）
```

**効果**: 3ケース → 1ショット

### 例2: 異常系の独立化

**テストケース**:
```
ケース10: 正常処理
ケース11: ファイルオープンエラー（リターンコード8）
```

**割り当て結果**:
```
ST007: ケース10（正常処理）
ST008: ケース11（ファイルオープンエラー・非0リターンコード発生）
```

**理由**: 異常終了ケースは独立したショットとする必要がある

### 例3: 複数パターンのグループ化

**テストケース**:
```
ケース13: ソートキー昇順パターン1
ケース14: ソートキー昇順パターン2
ケース15: ソートキー昇順パターン3
ケース16: ソートキー同値パターン
```

**割り当て結果**:
```
ST013: ケース13, 14, 15, 16（複数レコードで複数パターンを検証）
```

**効果**: 4ケース → 1ショット

## 使用例

```
/assign-testcase-shot
/assign-testcase-shot アウトプット/ABC0123_テストケース.tsv インプット/ABC0123_プログラム指示書.md
/assign-testcase-shot アウトプット/ABC0123_テストケース.tsv
```

## 注意事項

### 必須ルール

- **`read_file`前に必ず`list_files`で存在確認**
- **テストケースの意図を変更しない**
- **ショット割り当て案を必ずユーザーに提示し、承認を得る**
- **グループ化ルールに従う**
- **すべてのテストケースにショット番号を割り当てる**

### 禁止事項

- ユーザー承認なしに勝手にショット番号を割り当てること
- テストケースの内容を変更すること
- テストケースを省略すること

## 成果物形式

- **テストケース**: 元のTSVファイルを上書き更新（ショットNo列に新規割り当て）
