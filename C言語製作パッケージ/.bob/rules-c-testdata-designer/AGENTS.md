## 絶対遵守ルール（違反厳禁）

### 1. 入力ファイルの取り扱い

- `read_file` を実行する前に、必ず `list_files` でファイルの存在を確認すること
- 存在確認なしに `read_file` を呼び出すことは禁止

### 2. 前工程の承認確認

- **前工程（テストケース）が未承認の場合、作業を開始してはならない**
- 承認済みテストケースが TSV 形式で存在しない場合は、TSV 変換を先に実施するよう求め、処理を中断すること
- Markdown 形式のテストケースのみを入力として使用することは禁止

### 3. 成果物の形式

- **最終成果物は必ず TSV 形式で出力すること**
- Markdown 形式のまま最終成果物として提出することは禁止
- 中間成果物（レビュー・議論段階）の Markdown 提示は許可
- TSV ファイルは `成果物テンプレート/` 配下のテンプレートに従うこと
- テンプレート形式に従わない独自形式での提出は禁止

### 4. 出力先・ファイル名

- 出力先: `アウトプット/{PGM_ID}_テストデータ/`
- ファイル命名規則（厳守）:
  - 入力データ: `IN01.tsv`, `IN02.tsv`（2桁連番）
  - 出力データ: `OUT01.tsv`, `OUT02.tsv`（2桁連番）
- 上記以外のパス・ファイル名を使用することは禁止
- 入力データと出力データを同一ファイルに混在させることは禁止

### 5. テストケースとの対応

- 各テストデータは承認済みテストケースの ID に紐付けること
- **テストケースの意図を変更してはならない**
- **新規テストケースの追加は禁止**（前工程に戻る必要あり）
- すべての承認済みテストケース ID に対応するテストデータを生成すること

### 6. 入力情報の制限

- 入力は「PGM 指示書」＋「承認済みテストケース（TSV 形式）」＋「ファイル定義書」のみ
- **ソースコードへの言及は禁止**

### 7. 2段階プロセスの厳守

- テストデータ生成は必ず **Step 1（入力データ）→ Step 2（出力データ）** の順で実施すること
- Step 1 のユーザー承認を得る前に Step 2 へ進むことは禁止
- 各 Step は独立した TSV ファイルとして保存すること

### 8. ユーザー承認前の次工程移行禁止

- Step 1・Step 2 の両方がユーザーに承認されるまで、次工程（ソースコード生成）への移行を提案してはならない
- 次工程への移行を提案できるのは、以下がすべて揃った場合のみ:
  1. Step 1（入力データ）の AI チェック・修正が完了している
  2. Step 1 がユーザーに承認されている
  3. Step 2（出力データ）の AI チェック・修正が完了している
  4. Step 2 がユーザーに承認されている
  5. 入力・出力 TSV ファイルが `アウトプット/{PGM_ID}_テストデータ/` に保存されている
  6. テストケースとテストデータの対応表を提供している

### 9. AI チェックの実施タイミング

- 各 Step 完了後、かつユーザーレビューの前に、必ず第三者チェックを実施すること
- チェック省略は禁止

### 10. 仕様が曖昧な場合の対応

- 仕様が曖昧・不明確な場合は、推測で補完せずに「不明」として明示すること
- 仮定を置いた場合は、仮定内容と影響範囲を必ず記載すること
- 推測実装は禁止

## TSV ファイル形式ルール

### ヘッダー構造（4行固定）

```
行1: 空白3列 + 日本語項目名（タブ区切り）
行2: 空白3列 + C 言語変数名（タブ区切り）
行3: "使用" + "Case" + "Rec" + 型情報（C=char、I=int、L=long、F=float、D=double、P=pointer）
行4: "パターン" + "No." + "No." + サイズ + "コメント欄"
```

### データ行の構造

```
PT001   {テストケースNo}                        {説明}（0件ケース: データ行は空白行）
PT002   {テストケースNo}  1  {値1}  {値2}  ...  {説明}
PT003   {テストケースNo}  1  {値1}  {値2}  ...  {説明}（複数レコード: Rec No. を連番で記載）
        {テストケースNo}  2  {値4}  {値5}  ...  {説明}
```

### 形式ルール

- **パターン番号（PT001, PT002 ...）**: 各テストパターンの開始を示す
- **Case No. 列**: テストケース番号を記載
- **Rec No. 列**: 同一パターン内のレコード連番（1, 2, 3 ...）
- **0件ケース**: データ行を空白行のみとし、パターン行にコメントを記載
- **複数レコード**: 同一パターン内で Rec No. を連番で記載
- **コメント欄**: 各レコードの説明を記載

## AI 第三者チェックの義務

各 Step 完了後、ユーザーレビュー前に以下のプロセスを必ず実行すること:

1. 不足・誤りを特定する
2. 修正案を提示する（**修正理由を必ず明記すること**）
3. 修正後のテストデータを出力する
4. 修正内容のサマリーを提供する

### Step 1（入力データ）のチェック観点

- すべての承認済みテストケース ID に対応するデータがあるか
- テストケースの意図が正しく反映されているか
- ファイル命名規則（`IN01.tsv` 〜）に従っているか
- データ型が正しく指定されているか（C=char、I=int、L=long 等）
- 境界値・0件・複数件のパターンが明確に指定されているか
- 日付・時刻依存の処理は固定値を使用しているか（再現可能性の担保）
- 入力データと事前状態の整合性が取れているか

### Step 2（出力データ）のチェック観点

- 入力データとの ID 対応関係が正確か（PT 番号・Case No. が一致しているか）
- 期待結果の妥当性（PGM 指示書の処理仕様と一致しているか）
- ファイル命名規則（`OUT01.tsv` 〜）に従っているか
- データ整合性（参照整合性、マスタデータ存在前提が明示されているか）
- エラーコードの再現性（エラー処理の場合）

## セッション記憶ルール

[`.bob/rules/memory.md`](../rules/memory.md) に従い、以下を実施する:

### 開始時

- `MEMORY/knowledge/persistent.md` と `MEMORY/knowledge/preferences.md` を読み込む
- `MEMORY/reflections/` 配下を読み込む
- セッションログ新規作成: `MEMORY/sessions/YYYY-MM-DD_session-testdata-NNN.md`

### 終了時

- セッションログを完成させる
- エラーまたは新しい学びがある場合は `MEMORY/reflections/YYYY-MM-DD_reflection.md` を生成する
- `MEMORY/sessions/index.md` と `MEMORY/reflections/index.md` を更新する

---

## 違反時の対応

ルール違反を検出した場合:

1. **即座に作業を中断する**
2. **ルールに従った方法で再実行する**
3. **成果物を正しい形式で再生成する**
4. **チェックリストで再確認する**
