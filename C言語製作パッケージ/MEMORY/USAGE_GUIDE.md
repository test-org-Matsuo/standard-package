## セッション開始時の手順

### ステップ1: 必須ファイルの読み込み

```markdown
1. knowledge/persistent.md を読み込む
   - プロジェクト固有のルールを確認
   - 技術的な制約を把握
   
2. knowledge/preferences.md を読み込む
   - ユーザーのコミュニケーションスタイルを確認
   - ワークフロー設定を把握
```

### ステップ2: 最近のリフレクションの確認

```markdown
3. reflections/index.md を読み込む
   - 最新3件のリフレクションファイル名を確認
   
4. 最新3件のリフレクションを読み込む
   - 最近発生したエラーパターンを把握
   - 成功パターンを確認
   - 次回への引き継ぎ事項を確認
```

### ステップ3: タスク関連パターンの確認

```markdown
5. knowledge/patterns.md を読み込む（必要に応じて）
   - タスクに関連する成功パターンを確認
   - 避けるべき失敗パターンを確認
```

### 読み込み後のアクション

- 読み込んだ知識を基に、タスクのアプローチを計画
- ユーザーの好みに合わせたコミュニケーションスタイルを採用
- 過去の失敗パターンを避ける

## セッション中の記録方法

### セッションログファイルの作成

**ファイル名（正本参照）**: `sessions/YYYY-MM-DD_session-XXX-NNN.md`

**作成タイミング**: セッション開始時

**ルール**:
- 命名規則・採番規則・追記禁止ルールは `.bob/rules/memory.md` の定義に従う

**テンプレート**:
```markdown
# セッションログ: YYYY-MM-DD_session-XXX-NNN

## セッション情報
- **開始時刻**: YYYY-MM-DD HH:MM:SS JST
- **終了時刻**: （セッション終了時に記入）
- **タスク**: （タスクの概要）

---

## HH:MM - タスク開始

（ここから時系列で記録）

---

## 記録すべき重要事項

### 誤り
（セッション終了時にまとめる）

### 無駄
（セッション終了時にまとめる）

### 成功パターン
（セッション終了時にまとめる）

### 次回への引き継ぎ
（セッション終了時にまとめる）
```

### 記録すべき内容

#### ✅ 通常の作業
```markdown
## HH:MM - [作業内容の見出し]

**エージェントの対応**:
- 実施した作業の概要
- 使用したツール

**結果**: 成功/失敗
```

#### ❌ エラー発生時
```markdown
## HH:MM - [エラーの見出し]

**問題**: 
何が起きたか

**原因**: 
なぜ起きたか

**ユーザーのフィードバック**:
「ユーザーの指摘内容」

**エージェントの修正**:
- 修正内容1
- 修正内容2

**所要時間**: 約N分のロス
```

#### 💡 重要な発見
```markdown
## HH:MM - [発見の見出し]

**発見内容**:
何を発見したか

**影響**:
今後にどう活かせるか

**記録先**: persistent.md / patterns.md
```

## セッション終了時の手順

### ステップ1: セッションログの完成

```markdown
1. 終了時刻を記入
2. 「記録すべき重要事項」セクションを埋める
   - 誤り: 発生したエラーをリスト化
   - 無駄: 無駄だった作業をリスト化
   - 成功パターン: 効果的だったアプローチをリスト化
   - 次回への引き継ぎ: 重要な学びをリスト化
```

### ステップ2: sessions/index.md の更新

```markdown
3. 新しいセッションを表に追加
4. 統計情報を更新
   - 総セッション数
   - 平均所要時間
   - 平均エラー数
   - 総学習項目数
```

補足: `index.md` 更新必須条件は `.bob/rules/memory.md` に従う。

### ステップ3: リフレクションの生成判断

生成条件・生成不要条件は `.bob/rules/memory.md` を参照する。

### ステップ4: リフレクションの生成

**ファイル名（正本参照）**: `reflections/YYYY-MM-DD_reflection.md`  
命名・整合ルールは `.bob/rules/memory.md` に従う。  
内容は以下テンプレートに従う。

**テンプレート**:
```markdown
# セッションリフレクション: YYYY-MM-DD

## 📊 セッション概要
- **セッションID**: YYYY-MM-DD_session-XXX-NNN
- **タスク**: （タスク概要）
- **期間**: HH:MM-HH:MM (N時間)
- **ツール使用回数**: N回

## ❌ 発生した誤り

### 1. [誤りの名前]
- **発生箇所**: （どこで）
- **原因**: （なぜ）
- **影響**: （どんな影響）
- **対策**: （どう対策するか）
- **次回への教訓**: （次回どうするか）

## ⏱️ 無駄な作業

### 1. [無駄の名前]
- **内容**: （何が無駄だったか）
- **時間**: 約N分
- **改善策**: （どう改善するか）

## ✅ 成功パターン

### 1. [パターンの名前]
- **アプローチ**: （どんなアプローチ）
- **効果**: （どんな効果）
- **再現性**: 高い/中程度/低い

## 💡 次回セッションへの引き継ぎ

### プロジェクト固有ルール
1. （新しく発見したルール）

### ユーザーの好み・スタイル
1. （新しく分かった好み）

### 技術的な発見
1. （技術的な発見）

## 📈 改善メトリクス
- **エラー率**: N% (N/N)
- **リトライ率**: N% (N/N)
- **平均ツール実行時間**: N分
```

### ステップ5: reflections/index.md の更新

```markdown
5. 新しいリフレクションを表に追加
6. 統計情報を更新
```

### ステップ6: 知識ベースの更新

#### persistent.md の更新

**更新条件**:
- 新しいプロジェクト固有ルールが発見された
- 新しい技術的制約が発見された
- ユーザーの好みが明確になった

**更新方法**:
```markdown
1. 該当セクションに新しい項目を追加
2. 情報源とセッションIDを記録
3. 最終更新日を更新
```

#### patterns.md の更新

**更新条件**:
- 同じアプローチが3回成功した → 成功パターンとして記録
- 同じエラーが2回発生した → 失敗パターンとして記録
- 実験的アプローチの結果が出た → 試行錯誤として記録

**更新方法**:
```markdown
1. 該当セクションに新しいパターンを追加
2. 使用回数/発生回数を記録
3. 推奨度/重要度を設定
4. パターン統計を更新
```

#### preferences.md の更新

**更新条件**:
- ユーザーから明示的なフィードバックがあった
- ユーザーの好みが明確になった
- ワークフロー設定の変更があった

**更新方法**:
```markdown
1. 該当セクションを更新
2. 最終更新日を更新
```

## 記憶の更新ルール

この章の規範は `.bob/rules/memory.md` を正本とし、ここでは運用トリガーの早見表のみ示す。

### 自動更新のトリガー

| トリガー | 更新対象 | 更新内容 |
|---------|---------|---------|
| セッション開始 | sessions/YYYY-MM-DD_session-XXX-NNN.md | 新規セッションログを作成 |
| ツール使用 | sessions/[current].md | ツール使用記録を追記 |
| エラー発生 | sessions/[current].md | エラー詳細を追記 |
| ユーザーフィードバック | sessions/[current].md | フィードバック内容を追記 |
| セッション終了 | sessions/index.md | セッション情報を追加 |
| リフレクション生成 | reflections/index.md | リフレクション情報を追加 |
| 新ルール発見 | knowledge/persistent.md | ルールを追加 |
| パターン確立 | knowledge/patterns.md | パターンを追加/更新 |
| 好み変更 | knowledge/preferences.md | 設定を更新 |

## 実践例

### 例1: ファイル名誤りのセッション

#### セッション中の記録
```markdown
## 10:05 - ファイル名の誤り発生

**問題**: 
ファイル名を`SAMPLE.cbl`として生成しようとした。

**原因**: 
PGM-IDの確認を怠り、推測でファイル名を決定した。

**ユーザーのフィードバック**:
「ファイル名が間違っています。正しくはSAMPLE01.cblです」

**エージェントの修正**:
- PGM指示書を再確認し、PGM-ID = "SAMPLE01"を確認
- 正しいファイル名`SAMPLE01.cbl`で再生成

**所要時間**: 約5分のロス
```

#### リフレクション
```markdown
## ❌ 発生した誤り

### 1. ファイル名の誤認識
- **発生箇所**: SAMPLE.cbl → SAMPLE01.cbl
- **原因**: PGM指示書のPGM-ID読み取り不足
- **影響**: 1回のリトライ、5分のロス
- **対策**: PGM-ID確認を必須チェックリストに追加
- **次回への教訓**: ファイル名生成前に必ずPGM-IDを明示的に確認
```

#### 知識ベース更新

**persistent.md に追加**:
```markdown
### PR001: ファイル名規則
- **カテゴリ**: naming
- **ルール**: PGM-ID = ファイル名（拡張子除く）
- **情報源**: 2026-04-14セッション
- **信頼度**: 高
- **最終確認日**: 2026-04-14
```

**patterns.md に追加**（2回目の発生時）:
```markdown
### FP001: ファイル名の推測
- **説明**: PGM-IDを確認せずにファイル名を推測
- **結果**: ファイル名の誤り、リトライが必要、時間のロス
- **発生回数**: 2回
- **最終発生日**: 2026-04-14
- **防止策**: ファイル名生成前に必ずPGM-IDを明示的に確認
- **重要度**: ★★★★☆
```

### 例2: 成功パターンの確立

#### 3回目の成功時

**patterns.md に追加**:
```markdown
### SP001: 段階的DIVISION生成
- **説明**: IDENTIFICATION → ENVIRONMENT → DATA → PROCEDURE の順で生成
- **メリット**:
  - 構造が明確
  - エラー発生時の切り分けが容易
  - 各段階で確認が可能
- **使用回数**: 3回
- **成功率**: 100%
- **最終使用日**: 2026-04-14
- **推奨度**: ★★★★★
```
